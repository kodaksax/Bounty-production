# `profiles` SELECT RLS — Migration Strategy

> Written 2026-07-18. This is a **design document, not an implementation** — per the standing instruction not to blindly tighten this policy. It supersedes the "flagged, not fixed" status this finding has carried across three prior audits by actually cataloging every legitimate dependency first. This is app-wide in scope (not withdrawal-specific), filed here because it was surfaced during withdrawal-system auditing and the withdrawal docs already reference it repeatedly.

## The problem, precisely

Live policy (`supabase/migrations/20260303_add_profiles_rls_policies.sql:19-25`): `CREATE POLICY "profiles_select_authenticated" ... FOR SELECT TO authenticated USING (true)`. Comment on the policy states it's "needed for viewing posters/hunters" — a real, legitimate requirement, implemented in the broadest possible way: **any authenticated caller can read all 62 columns of any user's row**, including `balance`, `stripe_connect_account_id`, `stripe_customer_id`, `risk_level`, `risk_score`, `balance_frozen`, `email`, `phone`, `e2e_public_key`.

**Critical nuance: RLS is row-level, not column-level.** A policy can say "you may read this row" or "you may not," but it cannot say "you may read `username` on this row but not `balance`." That distinction matters for the design below — it rules out "just write a smarter RLS policy" as a complete solution on its own.

## Consumer catalog (condensed — see the full inventory in session notes for exact file:line detail)

**Legitimate cross-user reads needing only public-safe columns** (`username`, `avatar`, `display_name`, sometimes `location`/`about`): bounty listings and search (poster enrichment), conversation participant lists, blocked-user display, dispute-comment author display. ~10 call sites across `lib/services/bounty-service.ts`, `search-service.ts`, `supabase-messaging.ts`, `blocking-service.ts`, `dispute-service.ts`.

**Legitimate cross-user reads needing genuinely more**:
- `lib/services/e2e-key-service.ts` — reads `e2e_public_key` for another user, by design (it's meant to be public for encryption purposes, just not lumped in with truly sensitive fields).
- `app/profile/[userId].tsx` → `auth-profile-service.ts:getProfileById()` — currently `select('*')` (all 62 columns) to view **any** user's profile page, with a `public_profiles` view fallback only on error. The profile-view UI almost certainly doesn't need `balance`/Stripe fields/risk fields — needs verification against actual UI requirements before cutover, but is very unlikely to need more than the view's existing 8 columns plus perhaps 1-2 more (e.g. bio fields).
- `lib/services/user-search-service.ts` — `select('*')` across **all users**, and searches directly against the `email` column (`.or('email.ilike...')`). This is the single riskiest call site: full-column exposure *and* it's the reason a naive "just restrict columns" fix can't be purely mechanical — searching by email requires the search query to see the email column even though the results shouldn't expose it to the caller.
- `lib/services/bounty-request-service.ts` — `select('*')` batch fetch of hunter profiles for bounty-request lists.

**Admin panel — the load-bearing constraint.** `lib/admin/adminDataClient.ts` uses the **same anon-key client as the rest of the app** (`lib/supabase.ts`'s `supabase` export) — confirmed no `service_role` key anywhere in `app/(admin)/**`. `fetchAdminUsers()` and `fetchAdminUserById()` both do `select('*')`. **This means the admin panel's user-list and user-detail screens currently depend on the exact same overly-broad RLS policy this document is about restricting.** Any fix that doesn't separately provision admin access will break the admin panel outright. `app/(admin)/verifications.tsx` is already immune — it goes through a service-role edge function (`admin-verifications-list`), not a direct client query; that's the pattern to extend, not invent.

**`public_profiles` view already exists** — but only as untracked schema drift (not in any migration file), owner `postgres`, bypasses RLS via view-owner privilege. 8 columns: `id, username, display_name, avatar, location, about, verification_status, created_at`. Already used as a fallback in `auth-profile-service.ts` and `userProfile.ts`.

**Note for whoever implements this:** Supabase's security advisor flags this view as `ERROR`-level ("defined with the SECURITY DEFINER property") — confirmed live via `get_advisors`. The linter's generic remediation (set `security_invoker = true`) would be **wrong here**: it would make the view respect the querying user's own RLS on the base `profiles` table, which defeats the view's entire purpose as a curated safe-columns escape hatch. Don't apply the advisor's default fix blindly; the view needs to keep bypassing base-table RLS by design, since it's exposing a fixed, already-vetted-safe column set regardless of caller. Do formalize it as a tracked migration (step 1 below) so this intentional deviation from the linter's advice is documented, not silently ignored.

## Recommended mechanism: two complementary layers, not one

1. **Column-level `REVOKE`** on the truly sensitive columns (`balance`, `balance_on_hold`, `balance_frozen`, `stripe_connect_account_id`, `stripe_customer_id`, `stripe_connect_requirements`, `risk_level`, `risk_score`, `email`, `phone`, and any other risk/verification-only fields) from the `authenticated` and `anon` Postgres roles. This is a **structural, RLS-independent** guarantee — it works even if a future RLS policy has a bug, the same defense-in-depth principle already applied on the *write* side (`revoke_client_writes_on_sensitive_profile_columns` migration, already live). Column grants can't do per-caller dynamic checks (unlike RLS), so this uniformly blocks the columns for every `authenticated`/`anon` query — including the admin panel's, which is exactly why step 2 is required first.
2. **A new service-role-backed admin profile-read path**, matching the existing `admin-withdrawals`/`admin-review-id`/`admin-verifications-list` pattern exactly (JWT `app_metadata.role === 'admin'` check, service-role DB client, which bypasses both RLS and column grants). This is what lets the admin panel keep full-column access *after* step 1's REVOKE takes effect, without relying on RLS successfully distinguishing "admin" from "regular authenticated user" — which the existing RLS-policy-inconsistency finding (`auth.jwt()->>'role'` vs. `auth.jwt()->'app_metadata'->>'role'`, noted in a prior audit) suggests isn't reliably done elsewhere in this codebase yet either. Don't add a new RLS branch that repeats that same ambiguity.

The existing `USING (true)` row-level policy can stay largely as-is for the *row* dimension (any authenticated user may see that a row exists and its public columns) — the fix is almost entirely about *columns*, not rows.

## Migration steps, in safe (additive-first) order

1. **Formalize `public_profiles` as a tracked migration.** No behavior change — it already exists live; this just closes the "exists only as drift" gap consistent with this project's repeated lesson about untracked schema changes. Verify its 8 columns are suf ficient for `app/profile/[userId].tsx`'s actual UI needs first; add 1-2 more safe columns to the view if something legitimate is missing (e.g. a public bio field not yet in the 8).
2. **Build a new `admin-profiles` edge function** (or extend an existing admin function) with `list`/`getById` actions, service-role-backed, admin-JWT-gated — purely additive, doesn't touch the current broken-if-restricted path yet.
3. **Migrate `adminDataClient.ts`'s `fetchAdminUsers`/`fetchAdminUserById`** to call the new function instead of querying `profiles` directly. Verify the admin user-list and user-detail screens still work end-to-end before proceeding.
4. **Migrate remaining `select('*')` client call sites** (`getProfileById`, `user-search-service.ts`, `bounty-request-service.ts`) to use `public_profiles` or explicit safe-column selects instead of `select('*')`. For `user-search-service.ts`'s email search specifically: move it behind the same admin-only path (email search by an arbitrary user is arguably a separate, pre-existing privacy concern worth flagging on its own — verify whether this search feature is actually admin-gated in the UI today, since if it's callable by any authenticated user, that's a finding independent of the column-exposure issue).
5. **Only then apply the column-level `REVOKE`** on the sensitive-column list from `authenticated`/`anon`. This is the actual tightening step, done last, after every legitimate consumer has been verified not to depend on direct sensitive-column access anymore.
6. **Verify via a simulated-attack query** — as an authenticated non-owner, attempt to read `balance`/`email`/`stripe_customer_id` on another user's row directly; confirm it's rejected with a permission error. Matches the verification discipline already used for the write-side fix in a prior session.

## Rollback

Steps 1-4 are purely additive and independently revertible (new function/view, old code paths untouched until the explicit cutover in step 4). Step 5 (the `REVOKE`) is the only step with real blast radius if something was missed — have the exact `GRANT SELECT (...) TO authenticated` statement ready to run immediately if step 6's verification (or a post-deploy incident) reveals a consumer wasn't actually migrated.

## Why this wasn't implemented in this pass

The catalog above required a dedicated research pass to build (this session's Explore agent covered ~15 files); the actual migration touches the admin panel, a new edge function, and at least 4 client-side call sites, plus a genuine open question (is email search meant to be admin-only?) that needs a product decision, not just an engineering one. This is comparable in size to a full session on its own — consistent with every prior audit's conclusion that this needs dedicated scope, but this pass at least converts "flagged, not fixed" into an actionable, sequenced plan rather than repeating the same open question a fourth time.
