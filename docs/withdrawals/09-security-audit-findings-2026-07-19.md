## Fixed this session

### 1. Wallet balance didn't refresh after a withdrawal completed

**Root cause:** `components/withdraw-with-bank-screen.tsx` called `useWallet()`'s `refresh()` (reads the locally-cached SecureStore balance only) instead of `refreshFromApi()` (the one that actually re-fetches from the server) after a successful transfer. The withdrawal itself is a direct `fetch()` to `/connect/transfer`, not a wallet-context mutation, so the local cache was never updated by the withdrawal in the first place — calling the wrong refresher just re-displayed the stale pre-withdrawal number.
**User impact:** User withdraws funds, gets the "Withdrawal Initiated" success alert, returns to the Wallet tab, and sees their old (higher) balance until the app is backgrounded/foregrounded or the session token refreshes. No money was actually wrong — this was a display bug — but it's exactly the kind of thing that generates "did my withdrawal actually go through?" support tickets.
**Fix applied:** Changed the call to `refreshFromApi(session?.access_token)`. Since `WalletScreen` and `WithdrawWithBankScreen` share the same `useWallet()` context instance, this also fixes the Wallet tab's stale display on return — no separate fix needed there.
**Risk:** Low (display-only, one-line change, verified via `tsc`/`eslint` clean). **Blocks production deployment:** No, but should ship soon — real UX/support-load impact.
**Test coverage:** None added. `components/withdraw-with-bank-screen.tsx` has no existing component test to extend, and building one from scratch requires mocking session/wallet-context/fetch fairly extensively — flagged as a gap rather than rushed.

### 2. `fn_accept_bounty_request`'s authorization guard existed live but had zero corresponding git history

**Root cause:** A backend audit flagged this RPC as CRITICAL — `SECURITY DEFINER`, no caller-identity check, and the app's real/primary accept-bounty path (`lib/services/bounty-request-service.ts`) calls it directly via `supabase.rpc(...)`, bypassing the one Edge Function that does check poster ownership (which turns out to be unreachable in production anyway — see finding #4 below). Verified against the **live** function body (`pg_get_functiondef`, not the git file) before acting: production already has the exact guard needed (`IF auth.role() = 'authenticated' THEN ... bounty_row.poster_id <> auth.uid() ... RAISE EXCEPTION`), tracked in `schema_migrations` as `20260717232206` / `harden_fn_accept_bounty_request_authz` — but **no git file with that name, or any content match, has ever existed** (confirmed via `git log --all --grep` and a full `supabase/migrations/` listing). It was applied directly to production in a prior session and never backfilled into source control.
**User impact:** None currently — the vulnerability the audit was worried about is already closed live. The impact of the *drift* itself: a fresh environment built only from git (a disaster recovery restore, or any future Staging rebuild) would **not** have this authorization guard, silently reintroducing the CRITICAL hijack vector the live database no longer has.
**Fix applied:** `supabase/migrations/20260719010000_document_fn_accept_bounty_request_authz_guard.sql` — a pure `CREATE OR REPLACE`, transcribed from the verified live function definition, not from memory or the stale `20260421` git version. Not re-applied to production via `apply_migration` (blocked by the auto-mode safety classifier mid-session, see below) — since the content is byte-for-byte identical to what's already live, this is safe to leave as a git-only backfill for now; applying it is a no-op whenever it's next run.
**Risk:** None (documents existing behavior, doesn't change it). **Blocks production deployment:** No — production is already correct; this is a reproducibility fix.

## Found, NOT fixed — needs a decision or dedicated follow-up

### 3. `adminDataClient.updateUserStatus()` — admin ban/suspend/restore is likely non-functional for other users

**Root cause:** Calls `supabase.from('profiles').update({ status }).eq('id', id)` under the *admin's own* JWT (anon-key client, not service-role). Every UPDATE policy on `profiles` (`Users can update own profile` / `allow_profiles_update_self` / `profiles_update_own`) is `auth.uid() = id` — verified live via `pg_policy`, no admin-bypass policy exists anywhere. Separately, `profiles` has **no `status` column at all** (verified via a full `information_schema.columns` dump) — the closest real mechanism is `account_restricted`/`restriction_reason`/`restricted_at`, which nothing in `adminDataClient.ts` currently reads or writes, and a grep found zero references to `account_restricted` anywhere in `lib/`/`supabase/functions/`.
**User impact:** An admin using the panel's suspend/ban/restore action on another user's account almost certainly gets an explicit PostgREST error (unrecognized column), not a silent false-success as initially suspected — but either way, moderation actions on other users do not work today. A flagged/abusive account is not actually restricted.
**Risk:** High — this is a real gap in the platform's ability to act on abuse reports, though not itself an exploitable vulnerability (it doesn't grant unauthorized access, it just fails to restrict).
**Recommended fix:** Requires a product decision first (what should "suspend"/"ban"/"restore" actually set — `account_restricted` + `restriction_reason`? something new?), then a service-role-backed admin action (new `admin-profiles` action, or extend `admin-withdrawals`'s pattern) that performs the real column writes with an admin-role check, matching every other admin-panel-to-database path already fixed this month. **Not implemented this pass** — the schema-mapping question needs a human answer, not an engineering guess.
**Blocks production deployment:** No (pre-existing, not a regression), but worth prioritizing given it's a real trust-and-safety gap.

### 4. Bounty acceptance has two parallel, only-one-of-which-has-authz-checked-anywhere-visible code paths

`lib/services/bounty-request-service.ts`'s real path calls `fn_accept_bounty_request` directly (now confirmed safe live, see #2). Its own PGRST202 fallback (~lines 959-1059) reimplements "transition bounty to in_progress" + "reject competing requests" as three separate client round-trips with no explicit ownership check in the application code — **verified safe anyway**: `bounties_update_own`/`Owners can update their own bounties` RLS policies (`auth.uid() = poster_id`) already gate the actual mutation, so a non-poster's fallback attempt affects 0 rows rather than succeeding. Separately, `supabase/functions/accept-bounty-request/index.ts` (which *does* have an explicit ownership check in application code) is only called from a branch gated on `!isSupabaseConfigured` — which is always `false` in any real deployment (both env vars are mandatory for the app to function at all), so that Edge Function is effectively dead code.
**Risk:** Low as currently deployed (RLS is the real backstop for the fallback; the dead Edge Function isn't a live gap since the RPC itself is safe). Worth cleaning up: either delete the redundant fallback and Edge Function, or make the Edge Function the single real path and delete the direct-RPC call — not both. **Not implemented this pass** (architecture/ownership decision, not urgent). **Blocks production deployment:** No.

### 5. Stripe idempotency keys generated client-side but never sent to Stripe

`lib/services/apple-pay-service.ts` and the ACH/card/SetupIntent paths in `supabase/functions/payments/index.ts` generate/accept an `idempotencyKey` but never pass it to `stripe.paymentIntents.create(...)` (contrast with `connect/index.ts` and `wallet/index.ts`, which do this correctly). A client retry after a timeout can create a second, distinct PaymentIntent instead of Stripe deduplicating it.
**Risk:** Medium — orphaned PaymentIntents at minimum, a real double-charge risk if a caller ever confirms both. **Not implemented this pass** — touches live Stripe payment-intent creation paths, judged to need its own focused pass with test-mode verification (once Staging is usable) rather than a same-session drive-by fix. **Blocks production deployment:** No (pre-existing), but recommended as a near-term priority given the financial risk class.

### 6. Offline-queue bounty creation retry has no idempotency guard

`lib/services/offline-queue-service.ts`'s retry-with-backoff wrapper is sound, but the bounty-creation path it retries (`bountyService.processQueuedBounty()`) does a bare insert with no idempotency key or unique constraint — unlike the online creation path, which has its own in-process dedupe wrapper that `processQueuedBounty` doesn't go through. A success-but-client-never-saw-the-response scenario (exactly what the offline queue exists for) can insert a duplicate bounty with duplicated escrow funding on retry.
**Risk:** Medium, real money (duplicated escrow), but requires a specific network-timing failure to trigger. **Not implemented this pass. Blocks production deployment:** No (pre-existing), recommended as a near-term priority alongside #5.

### 7. Profile-hook fragmentation has a real, currently-shipping consequence

Five overlapping profile hooks/services (`useAuthProfile`/`auth-profile-service.ts`, `useProfile`/`user-profile-service.ts`, `useUserProfile`/`userProfile.ts`, `useNormalizedProfile`, `use-auth-context`/`auth-provider.tsx`) exist as competing, non-synchronized sources of truth, not genuinely different needs. Concretely: `components/skillset-edit-screen.tsx` saves through `useUserProfile`, which writes **only to `AsyncStorage`**, never to Supabase — so skill edits are visible on the editor's own profile tab (which happens to read from the same local cache key) but never reach the backend, meaning no other user, and no fresh install, ever sees them, despite a "Skills saved" success toast.
**Risk:** High for this specific flow (silent, permanent-until-noticed data loss from the user's perspective), medium for the broader duplication (ongoing maintenance/inconsistency risk, and the likely root cause of why a still-live screen picked the wrong hook in the first place). **Not implemented this pass** — consolidating five hooks into one source of truth is a substantial refactor better suited to its own session. **Blocks production deployment:** No (pre-existing), but recommended as a near-term priority — this is actively losing user data today.

### 8. Confirmed dead code (frontend)

A reference-count scan of `components/` found roughly two dozen components with zero imports anywhere in `app/`, `components/`, `hooks/`, `lib/`, or `providers/` — a self-documented `@deprecated` 650-line profile-edit screen, a whole unused React-web admin-dashboard template cluster, several superseded onboarding/security screens, and ~20 dead shadcn/ui web primitives that don't apply to this Expo/React Native app. Also: `lib/feature-flags.ts`'s `FOLLOW_FEATURE_ENABLED` is a hardcoded `false` gating a "follow" feature whose backing service (`lib/services/follow-service.ts`) is entirely in-memory mock data, never wired to a real table.
**Risk:** Low (no runtime impact — dead weight, contributor-confusion risk, the kind of thing that caused #7's skillset bug in the first place). **Not implemented this pass** (large deletion, wanted a human confirm-and-search pass first per this project's stated preference for reversible steps over unilateral deletion of possibly-in-progress work). **Blocks production deployment:** No.
